# deployment

**Purpose:** Config/secrets split (12-factor) plus the macOS `launchd` runtime that keeps the processor, publish poller, and (optional) publish webhook running — wraps Anaconda Python, loads `.env`, and manages restart/crash-loop behavior.

- `~/Library/LaunchAgents/com.productcoffee.firstcup.plist` (INSTALLED — **authoritative, verified 2026-07-14**) → what launchd actually runs: `ProgramArguments` = **`run_processor.sh`** (single arg), **`KeepAlive: true`**, **`RunAtLoad: true`**, **no `WatchPaths`** (file pickup is the in-process `watch_directory` poll loop, not launchd). - `com.productcoffee.firstcup.poller.plist` → publish-poller Launch Agent — `Label: com.productcoffee.firstcup.poller`; runs `/Users/jasonbrett/anaconda3/bin/python3 publish_poller.py`; `RunAtLoad: false` (spawned on-demand by `youtube_processor.py`, not by launchd); `KeepAlive: false` (self-terminates after 24h or on publish success); logs to `logs/poller_stdout.log` / `logs/poller_stderr.log`.
- `com.productcoffee.firstcup.webhook.plist` → optional webhook-server Launch Agent — `Label: com.productcoffee.firstcup.webhook`; runs `publish_webhook.py`; **`RunAtLoad: true` AND `KeepAlive: true`** (the only one of the three plists set up as an always-on daemon); logs to `logs/webhook_stdout.log` / `logs/webhook_stderr.log`.
- `config.json` / `config.json.template` → non-secret settings — `directories.{transcripts,outputs}`, `templates.{youtube_description,newsletter_examples}`, `api.{model,watch_interval}`, `slack.enabled`. **`api.model` IS the live model source of truth** — read via `model_registry.get_model()` on every API call (as of 2026-07-15; it was decorative before then). `directories.*` and `templates.*` are read directly by `youtube_processor.py`; `api.watch_interval` is still unused (the loop uses the `WATCH_INTERVAL` constant).
- `.env.template` → secrets contract (key names only) — `ANTHROPIC_API_KEY`, `SLACK_WEBHOOK_URL`, `SLACK_BOT_TOKEN`, `SLACK_USER_ID`, `YOUTUBE_API_KEY`, `WP_SITE_URL`, `WP_USERNAME`, `WP_APP_PASSWORD`. `.env` itself is gitignored (`.gitignore`: `.env`, `*.env`, `logs/`, `*.log`).
- `model_registry.py` → `get_model()` / `set_model()` — the model funnel. `get_model()` resolves `ANTHROPIC_MODEL` env > `config.json api.model` > `DEFAULT_MODEL`; `set_model()` persists a choice (read-modify-write). `DEFAULT_MODEL` + `CURATED_FALLBACKS` are the ONLY hardcoded model literals allowed anywhere (enforced by `test_model_registry.py`). `youtube_processor.load_config()` back-fills a missing `api.model` from `DEFAULT_MODEL`.
- `logs/stdout.log`, `logs/stderr.log`, `logs/poller_stdout.log`, `logs/poller_stderr.log`, (`logs/webhook_stdout.log`, `logs/webhook_stderr.log` if webhook agent installed) — plain files, launchd redirects the child process's fds here; not rotated (LAUNCH_AGENT_GUIDE.md documents manual `> logs/stdout.log` truncation).

## Key flows
- **Cold start / restart (production path, verified):** `launchctl load ~/Library/LaunchAgents/com.productcoffee.firstcup.plist` → launchd (`RunAtLoad: true`) execs **`run_processor.sh`** → wrapper `cd`s to repo, `set -a; source .env; set +a`, then `exec`s Anaconda python on `youtube_processor.py`. Because of `exec`, the shell is REPLACED by python — so `ps` shows `/Users/jasonbrett/anaconda3/bin/python3 youtube_processor.py` with no shell parent. Don't let that fool you into thinking `run_processor.sh` wasn't involved: it was, and it is what sourced `.env`.
- **File pickup is NOT launchd `WatchPaths`** on the installed agent — it's the in-process `watch_directory` poll loop in `youtube_processor.py` (`WATCH_INTERVAL = 10s`), comparing `transcripts/*.txt` against `outputs/.processed_transcripts.json`. The committed template's `WatchPaths` is not in effect.
- **Install/update:** edit `com.productcoffee.firstcup.plist` template → `./install_launch_agent.sh` re-substitutes placeholders and reloads → verify with `launchctl list | grep firstcup`.
- **Restart procedure (memory-documented shorthand "restart the background server"):** `launchctl unload ~/Library/LaunchAgents/com.productcoffee.firstcup.plist; launchctl load ~/Library/LaunchAgents/com.productcoffee.firstcup.plist`.
- **On-demand child processes:** `youtube_processor.py` spawns `publish_poller.py` after a transcript finishes processing; that poller's own Launch Agent (`com.productcoffee.firstcup.poller.plist`) exists only so launchd can supervise/log it — `RunAtLoad: false` means launchd itself never starts it standalone.

## Mechanism notes  ← HIGHEST SIGNAL
- **`.env` shell-quoting footgun (#1 risk).** `run_processor.sh` loads secrets via `set -a; source .env; set +a` — this is bash *sourcing* the file, not a safe key=value parser. Any secret value containing spaces or shell metacharacters MUST be quoted in `.env`, or bash truncates the assignment at the first space and tries to run the remainder as a command. This exact bug took down WordPress publishing: an unquoted `WP_APP_PASSWORD` (WordPress app passwords are space-separated, e.g. `abcd 1234 efgh 5678`) got truncated at the first space, spawning phantom commands (`xMg7: command not found` spam in `logs/stderr.log`) while auth silently failed downstream in `blog_publisher.py`. Now fixed by quoting. **Any new `.env` value with spaces must be quoted** — this is the standing risk for every future credential added to `.env.template`.
- **Model source of truth (RESOLVED 2026-07-15 — was a trap, now the fix).** Historically `config.json api.model` was decorative and the real value was a hardcoded `MODEL` constant, so a retired ID went stale unnoticed and 404-crash-looped for weeks. That's fixed: `api.model` is now authoritative via `model_registry.get_model()`, the constant is gone, and a retired model triggers the Slack-driven `ModelUnavailableError` recovery instead of a crash-loop. To change the model, edit `config.json api.model` (or `ANTHROPIC_MODEL`), or just reply in Slack when prompted. A build-failing test blocks any new hardcoded literal.
- **stdout is block-buffered when redirected to a file, so `logs/stdout.log` lags.** Because launchd redirects the child's stdout to a plain file (not a TTY), Python's default buffering switches from line-buffered to block-buffered — `tail -f logs/stdout.log` can look frozen or empty for long stretches even while the process is actively working through the pipeline. Neither the plist nor `run_processor.sh` sets `PYTHONUNBUFFERED=1` or invokes `python3 -u`, so this is unmitigated. Anyone debugging via `tail stdout.log` should not conclude "hung" from silence — check `logs/stderr.log` and process liveness (`launchctl list | grep firstcup`, `ps`) instead of trusting stdout timing.
- **INSTALLED PLIST HAS DRIFTED FROM THE COMMITTED ONE — trust `~/Library/LaunchAgents/`, not the repo.** Verified 2026-07-14: the installed agent runs `run_processor.sh` with `KeepAlive: true` / `RunAtLoad: true` / no `WatchPaths`; the committed `com.productcoffee.firstcup.plist` still holds unsubstituted `*_PLACEHOLDER` strings and describes a different runtime entirely (`/usr/bin/python3`, `KeepAlive: false`, `WatchPaths`). Anyone reasoning from the repo file will draw wrong conclusions about how the service starts, whether `.env` is loaded, and whether it respawns. Re-running `install_launch_agent.sh` would OVERWRITE the working installed agent with the stale template — treat that script as dangerous until the template is reconciled.
- **`KeepAlive: true` on the installed main agent turns any hard failure into an infinite crash-loop.** This is not theoretical: when the model ID went stale (retired `claude-sonnet-4-20250514` → 404 on every call, before model resolution moved into `model_registry`), launchd respawned the processor endlessly and it retried the same transcript forever, filling `logs/stdout.log` with identical 404s for weeks. (The 404-specific case is now handled by `ModelUnavailableError`; `KeepAlive` still amplifies *other* hard failures the same way.) Signature: the same error + same `request_id` repeating, and a changing PID in `launchctl list | grep firstcup`. `KeepAlive: true` means the service never dies loudly — it fails quietly and forever, so absence of a crash is NOT evidence of health.
- **`com.productcoffee.firstcup.webhook.plist`** also sets `KeepAlive: true` + `RunAtLoad: true` (always-on daemon) — same respawn-loop exposure if `publish_webhook.py` fails at bind/import. Check `logs/webhook_stderr.log` for rapid repeated timestamps.
- **Hardcoded Anaconda interpreter path** (`/Users/jasonbrett/anaconda3/bin/python3`) in `run_processor.sh` and in the poller/webhook plists — machine-specific; fails with "no such file" on any other machine or if Anaconda moves. Dependencies (`anthropic`, `requests`, and the WIP `google-genai`/`Pillow`) are installed into that Anaconda env, so the committed template's `/usr/bin/python3` would `ImportError` on `anthropic` even if it did run — another reason the stale template must not be installed.
- **Restart is the universal fix-it lever**: `launchctl unload` + `launchctl load` of the relevant plist picks up code and `config.json` changes (env var / `.env` changes require this too, since `.env` is only re-sourced at process start via `run_processor.sh`, if that path is the one actually invoked).

## Related subsystems
- **processor** (`youtube_processor.py`) — the process this deployment layer starts/supervises; owns `load_config()` and resolves the model via `model_registry.get_model()`.
- **publishing** (`blog_publisher.py`, `publish_poller.py`) — poller is spawned on-demand by the processor and supervised by `com.productcoffee.firstcup.poller.plist`; consumes `WP_*`/`YOUTUBE_API_KEY` secrets from the same `.env`.
- **slack** (`slack_helper.py`) — reads `SLACK_*` secrets from the same `.env`; `config.json`'s `slack.enabled` flag gates whether it's used at all.
