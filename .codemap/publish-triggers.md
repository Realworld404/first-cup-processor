# publish-triggers

**Purpose:** Two independent, unsynchronized mechanisms that detect a "publish this episode" signal and call into `blog_publisher.publish_first_cup()` to create a WordPress draft — a spawned background poller (Slack reactions/replies) and a standalone HTTP webhook server.

## Key files
- `publish_poller.py` → `run_poller`, `load_state`, `delete_state`, `check_for_emoji_reaction`, `check_for_publish_reply`, `send_slack_notification` — subprocess spawned per-episode; polls Slack every 60s for a 📤 reaction or "publish" reply, then calls `blog_publisher.publish_first_cup`.
- `publish_webhook.py` → `PublishHandler.handle_publish`, `run_server` — always-on `BaseHTTPRequestHandler` on port 8765; `POST /publish` calls `blog_publisher.find_most_recent_output()` then `publish_first_cup`, no per-episode targeting.
- `youtube_processor.py` → `spawn_publish_poller`, `process_transcript_file` — spawns the poller via `subprocess.Popen(..., start_new_session=True)` right after `slack.notify_completion()`, once per processed transcript.
- `slack_helper.py` → `SlackHelper.save_poller_state`, `SlackHelper.get_completion_message_info`, `SlackHelper.notify_publish_success`, `SlackHelper.notify_publish_error` — writes the state file the poller reads; called once per transcript, always to the same fixed path.
- `blog_publisher.py` → `publish_first_cup`, `find_most_recent_output`, `BlogPublisher.create_post` — the single publish path both triggers converge on; no dedup/idempotency check before creating the WP post.
- `.publish_poller_state.json` (repo root, gitignored) → single JSON blob: `channel`, `message_ts`, `thread_ts`, `output_path`, `selected_title`, `created_at`. Path is a hardcoded constant (`DEFAULT_STATE_FILE` / `script_dir / ".publish_poller_state.json"`), **not** per-episode.
- `com.productcoffee.firstcup.poller.plist` → `RunAtLoad=false`, `KeepAlive=false` — never launchctl-managed at boot; exists purely so launchd *could* supervise it, but in practice the poller is started directly by `subprocess.Popen`, not by `launchctl load`.
- `com.productcoffee.firstcup.webhook.plist` → `RunAtLoad=true`, `KeepAlive=true`, port 8765 — this one *is* meant to run continuously under launchd (if installed/loaded).

## Key flows
- **Poller path:** `process_transcript_file` (youtube_processor.py) → `slack.notify_completion()` → `spawn_publish_poller(slack, output_path, selected_title)` → `slack.save_poller_state(state_file)` overwrites `.publish_poller_state.json` → `subprocess.Popen([...publish_poller.py, --state, state_file], start_new_session=True)` → poller loop polls `reactions.get` / `conversations.replies` every 60s → on match, `publish_first_cup(output_path, selected_title)` → Slack success/error message → `delete_state(state_file)` → process exits.
- **Webhook path:** external POST to `http://localhost:8765/publish` → `PublishHandler.handle_publish` → `find_most_recent_output()` (picks newest-mtime dir under `outputs/`, ignoring the state file entirely) → `publish_first_cup(output_dir)` → Slack notify via a *fresh* `SlackHelper()` instance (no thread context) → JSON response.
- Both paths terminate in `blog_publisher.create_post()`, which always creates a new WordPress draft post — there is no check for "was this episode already published."

## Mechanism notes  ← HIGHEST SIGNAL
- **Global, single-slot state file — classic overwrite race.** `.publish_poller_state.json` is one fixed path, not namespaced per episode/output_path. If a second transcript finishes processing (and a second poller is spawned) before the first poller's episode has been published/timed out, `save_poller_state` silently overwrites the file the *first* poller has already loaded into memory — but the first poller only reads the file once at startup (`load_state` in `run_poller`), so it keeps polling with its **stale in-memory `channel`/`message_ts`/`thread_ts`/`output_path`** while the file on disk now points at episode 2. Net effect: a 📤 reaction on episode 2's Slack message could cause poller #1 (still running, still watching episode 1's `message_ts`) to publish **episode 1** when the user meant to trigger episode 2, or vice versa depending on timing. There is no lock, no per-PID/per-episode filename, no read-modify-write guard.
- **State file deletion as a footgun for concurrent pollers.** Each poller's `run_poller` loop checks `if not os.path.exists(state_file): shutdown` (manual cancellation) using the *same shared path*. If poller #1 finishes and calls `delete_state(state_file)` while poller #2 is still using that same filename (because state was overwritten, not per-episode), poller #2 will see the file vanish and shut down without publishing — silent cross-episode cancellation.
- **Orphaning risk on parent death:** `spawn_publish_poller` uses `start_new_session=True` (detaches from youtube_processor's process group/session), so if `youtube_processor.py` (the watch-mode parent) crashes or is killed, the poller subprocess survives as an orphan — this is actually the *intended* behavior (poller must outlive the watcher across the 24h window), not a bug. But it also means there's no supervisory mechanism to notice a poller that itself crashes mid-poll (e.g., uncaught exception outside the `try/except KeyboardInterrupt` in `run_poller` would traceback and die silently since stdout/stderr are `DEVNULL` at spawn time — actual poller stdout/stderr goes to `logs/poller_stdout.log`/`poller_stderr.log` only if launched via the plist, NOT when spawned via `subprocess.Popen` with `stdout=DEVNULL, stderr=DEVNULL` — meaning poller crashes during normal (non-launchd) operation are invisible, nothing is logged anywhere).
- **Rate limits / poll cost:** 60s interval × 2 Slack API calls per poll (`reactions.get` + `conversations.replies`) × up to 24h (1440 polls → up to 2880 calls) per episode. Multiple simultaneous pollers (one per episode, if several transcripts are processed same-day) multiply this linearly against the same bot token — plausible source of Slack API rate-limit errors if First Cup processes several episodes in a short window, compounding the state-file race above (more concurrent pollers = higher chance of an overwrite mid-flight).
- **Double-publish risk between poller and webhook:** the two trigger paths are fully independent and never check each other. `publish_first_cup`/`create_post` has no idempotency guard (no "already published" marker written to the output directory or state file). If the webhook is hit (`curl -X POST /publish`) while a poller is *also* about to fire for the same episode (e.g., user both curls the webhook AND reacts with 📤), two separate WordPress draft posts get created for the same episode — nothing prevents this. `find_most_recent_output()` (webhook) has no relationship at all to `output_path` in the state file, so the webhook could target a *different* (newer) episode than any currently-polling poller believes it owns — another cross-episode confusion vector.
- **Webhook has zero auth:** `publish_webhook.py`'s `PublishHandler.do_POST` accepts any POST to `/publish` with no token/secret check — if port 8765 is reachable beyond localhost (e.g., firewall misconfig, `''` bind address `server_address = ('', port)` binds all interfaces, not just `127.0.0.1`), any network client can trigger a real WordPress publish.
- **Emoji reaction detection scope dependency:** `check_for_emoji_reaction` calls `reactions.get`, which requires the `reactions:read` bot scope (documented in CLAUDE.md); if that scope is missing/revoked, `reactions.get` will return `data.get("ok") == False` with some error other than `"no_reaction"`, which is logged as a warning but the poller keeps running — meaning the 📤 path silently never triggers while the "publish" text-reply path continues to work (partial/degraded operation, not a hard failure).
- **TTL cleanup on timeout is clean but content-blind:** hitting the 24h `timeout_at` calls `delete_state(state_file)` and exits — since the state file is global/shared, this deletion could also kill a *different*, more-recently-spawned poller that inherited the same filename (see overwrite race above), terminating it silently well before its own 24h window.

## Related subsystems
- **blog_publisher.py** (publishing) — the convergence point for both triggers; owns WordPress REST calls, YouTube thumbnail/video lookup (`get_youtube_video_for_title`, `get_most_recent_video`), and `BlogPublisher.create_post`.
- **slack_helper.py** (slack) — owns thread state (`thread_ts`, `get_completion_message_info`) that seeds the poller's state file; also used independently by the webhook via a fresh `SlackHelper()` with no thread continuity.
- **youtube_processor.py** (processor) — the only spawner of the poller; owns the per-transcript processing pipeline and decides when `spawn_publish_poller` fires.
- **Launch Agent / deployment** — `com.productcoffee.firstcup.plist` (main watcher), `com.productcoffee.firstcup.poller.plist` (dormant/unused-in-practice), `com.productcoffee.firstcup.webhook.plist` (the only always-on trigger surface if installed).
