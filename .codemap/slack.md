# slack

**Purpose:** Slack integration layer for the First Cup pipeline — posts progress notifications, collects interactive human decisions (title selection, thumbnail selection, publish trigger, credits-exhaustion resume) via threaded conversations, and exposes non-blocking checks for background pollers.

## Key files
- `slack_helper.py` → `SlackHelper` — single class holding all Slack state and API calls; optional subsystem (`is_enabled()` gates on webhook_url + bot_token + user_id all being set).
- `slack_helper.py` → `start_new_thread`, `send_message`, `send_webhook` — thread lifecycle + message senders (bot-token vs. webhook paths).
- `slack_helper.py` → `send_title_options`, `poll_for_response` — Step 1 title selection request/response cycle.
- `slack_helper.py` → `notify_credits_exhausted`, `poll_for_resume` — credits-exhaustion pause/resume flow (top-level alert + blocking wait for "resume").
- `slack_helper.py` → `notify_completion`, `check_for_publish_command`, `check_for_emoji_reaction`, `get_completion_message_info`, `save_poller_state` — post-completion publish-trigger support (both inline check and handoff state for the separate `publish_poller.py` process).
- `slack_helper.py` → `upload_file`, `send_thumbnail_options`, `poll_for_thumbnail_response` — thumbnail review flow (file upload + numbered/feedback selection, same polling pattern as titles).
- `slack_helper.py` → `notify_processing_start`, `notify_title_selected`, `notify_generating_new_titles`, `notify_error`, `notify_cancelled`, `notify_publish_started`, `notify_publish_success`, `notify_publish_error`, `test_connection` — one-shot status notifications, all threaded (`in_thread=True` default) via `send_message`.

## Key flows
- **Title selection:** `youtube_processor.py` calls `start_new_thread()` → `notify_processing_start()` → `send_title_options()` (sets `self.thread_ts` from first message's `ts`) → `poll_for_response()` blocks until user replies with a number / `f feedback` / `TITLE: ...` in-thread.
- **Credits exhaustion:** on `APICreditsExhaustedError`, processor calls `notify_credits_exhausted(filename)` (deliberately `in_thread=False` — top-level ping so it's not buried) → `poll_for_resume()` blocks on the *new* thread (its own `thread_ts`, set by that top-level message) until a reply of exactly `"resume"` arrives, then processing retries the same file.
- **Completion → publish:** `notify_completion()` stores `last_output_path`; either (a) foreground: `check_for_publish_command()` is polled by the caller during idle time and directly invokes `self.publish_callback(output_path, selected_title)` set via `set_publish_callback()`, or (b) background: `save_poller_state()` dumps thread/channel/output info to a JSON file so the separate `publish_poller.py` process can independently detect a 📤 reaction (`check_for_emoji_reaction`) or a "publish" reply and drive WordPress publishing itself.
- **Thumbnails:** `send_thumbnail_options()` posts the prompt then `upload_file()`s each PNG in-thread; `poll_for_thumbnail_response()` mirrors `poll_for_response()`'s loop/parsing structure for select / `N: feedback` / `regenerate all` / `done`.

## Mechanism notes ← HIGHEST SIGNAL
- **Threading invariant:** `self.thread_ts` is set ONLY the first time `send_message()` succeeds after `start_new_thread()` reset it to `None` (the `if not self.thread_ts: self.thread_ts = data.get("ts")` branch). Every subsequent `send_message(..., in_thread=True)` call attaches `thread_ts` to keep replies in one thread. `self.last_message_ts` is a *different* value (the most recent message's own ts, used for reaction lookups) — code must never substitute one for the other, per CLAUDE.md's explicit warning.
- **Hard blocking calls — these stall the caller's whole process:** `poll_for_response()` (title/thumbnail selection) and `poll_for_resume()` (credits) both run `while True: ... time.sleep(poll_interval)` with no default timeout (`timeout_seconds=None` is the default for `poll_for_response`; `poll_for_resume` has no timeout parameter at all — it waits forever). Since `youtube_processor.py`'s watch loop calls these synchronously, the entire watcher is frozen until a human replies in Slack. `poll_for_thumbnail_response()` has the same shape.
- **Non-blocking counterparts exist for background use:** `check_for_publish_command()` and `check_for_emoji_reaction()` each make a single HTTP GET and return immediately (`return False` on any exception — "silent fail" is intentional here since these run inside idle-loop polling, not as the main flow).
- **Robust JSON handling:** every `response.json()` call across the file is wrapped in `try/except ValueError` (see `send_message`, `poll_for_response`, `poll_for_resume`, `check_for_publish_command`, `check_for_emoji_reaction`, `upload_file`, `poll_for_thumbnail_response`). This was a deliberate fix (per CLAUDE.md) — a malformed/HTML error body from Slack used to raise unhandled and crash the service; now it's logged (or silently skipped for the background-check functions) and the loop continues.
- **Dedup via `processed_messages` set:** all three poll loops (`poll_for_response`, `poll_for_resume`, `poll_for_thumbnail_response`) track seen `ts` values locally per-call to avoid reacting twice to the same reply on repeated `conversations.replies` fetches — this set is NOT persisted, so a process restart mid-poll would need to reprocess from the API but has no memory of what was already handled (not currently an issue since these are long-lived blocking calls, not resumable).
- **Poll interval:** fixed 5s for title/thumbnail polling, 30s default for `poll_for_resume` (credits are expected to take longer to resolve — human has to go add billing funds).
- **Shared mutable state on `SlackHelper` instance** (no locking — single-threaded assumption): `thread_ts`, `last_message_ts`, `last_output_path`, `last_selected_title`, `publish_callback`. `check_for_publish_command()` clears `last_output_path`/`last_selected_title`/`thread_ts` after a successful publish to prevent double-publishing on next idle poll.
- **Auth/config:** `SLACK_BOT_TOKEN`, `SLACK_USER_ID`, `SLACK_WEBHOOK_URL` env vars; `is_enabled()` requires all three of webhook_url + bot_token + user_id — missing any one silently disables Slack (callers must check `is_enabled()`/handle `None` returns rather than assume Slack is present). Required bot scopes per CLAUDE.md: `chat:write`, `channels:history`, `groups:history`, `reactions:read`.
- **Two send paths:** `send_webhook()` (simple, unthreaded, webhook-only) is effectively unused by the interactive flows — everything that needs threading goes through `send_message()` (bot-token based, supports `thread_ts`). `send_webhook` exists as a fallback/simple-notify path.

## Related subsystems
- **processor** (`youtube_processor.py`) — owns the watch loop, calls into `SlackHelper` for every step (title gen, credits-exhaustion pause/resume, completion), and is the thing that gets blocked by the polling calls.
- **publish-triggers** (`publish_poller.py`) — separate background daemon that consumes `save_poller_state()`'s JSON file and independently re-implements reaction/reply detection against the same Slack thread, so it can keep watching for 24h after the main processor moves on.
- **publishing** (`blog_publisher.py`) — the actual WordPress/YouTube work invoked via `publish_callback`; `SlackHelper` only orchestrates the notify-before/after (`notify_publish_started`, `notify_publish_success`, `notify_publish_error`) around it.
